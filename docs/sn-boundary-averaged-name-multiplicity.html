§1 — Problem (fully defined)

The DD subtree summary/local/separatrix_average/* stores flux-surface-averaged quantities evaluated at the separatrix (the plasma boundary). Its StandardNames are a multiplicity mess: of 54 dd-sourced leaves, 24 carry more than one live name. Three coexisting forms were composed for the same physical quantity, and all are name_stage='accepted':

  • Long form — flux_surface_averaged_<q>_at_plasma_boundary (e.g. flux_surface_averaged_carbon_density_at_plasma_boundary).
  • Short form — <q>_at_plasma_boundary (e.g. carbon_density_at_plasma_boundary). 14 leaves carry BOTH the long and short form — a straight duplicate pair.
  • Locus-stripped — no _at_plasma_boundary at all (e.g. flux_surface_averaged_carbon_density, lithium_density). 6 leaves carry a locus-stripped name. Some are strict duplicates of a with-locus sibling; others (e.g. flux_surface_averaged_boron_density, flux_surface_averaged_total_ion_density, flux_surface_averaged_electron_temperature, flux_surface_averaged_magnetic_shear) are the only name for that leaf and simply lack the locus. One degenerate exhausted name (flux_surface_averaged_velocity, no species) also exists.

§1a — Correction: the operator and the locus are orthogonal (2026-07-10)

This plan's original framing — "the plasma boundary IS a flux surface, so the flux_surface_averaged_ prefix is redundant with _at_plasma_boundary" — was wrong in general, in both directions:

  • The locus is never redundant. The plasma boundary is one surface in a continuum of nested flux surfaces; _at_plasma_boundary pins which surface. Every name in this family keeps the locus. The locus-stripped names are wrong in all cases.
  • The operator is not implied by the locus. A quantity evaluated at the plasma boundary can still vary along that surface (poloidal/toroidal asymmetry). The DD encodes exactly this distinction with two parallel 64-leaf subtrees: summary/local/separatrix/* (the local value) and summary/local/separatrix_average/* (the flux-surface average). A locus-only name cannot denote both.
  • The redundancy is parameter-dependent. It exists only for bases that are constant on a flux surface (flux functions): the safety factor is already a flux-surface quantity (averaging it is an FSA of an FSA), magnetic shear derives from the q profile, psi/rho_tor are flux labels, and pressure is a flux function in static ideal MHD. For these, FSA is a no-op: the local and averaged DD leaves are the same physical quantity and should share ONE name (two sources → one StandardName). For surface-varying bases (rotation velocities, momentum, parallel electric field, impurity densities with centrifugal poloidal asymmetry), the prefix carries physical meaning and stays.

Graph evidence (2026-07-10): 54 separatrix_average leaves carry live names (24 with 2–3); 30 separatrix (local) leaves carry names; and 15 names are live on BOTH subtrees. For flux labels this sharing is correct (poloidal_magnetic_flux_at_plasma_boundary, toroidal_flux_radius_at_plasma_boundary, safety_factor_at_normalized_poloidal_magnetic_flux_equal_to_0_95); for surface-varying bases it is a collision (e.g. toroidal_argon_velocity_at_plasma_boundary, toroidal_plasma_momentum_at_plasma_boundary, iron_density_at_plasma_boundary) — the name can no longer distinguish the local value from the average. An anomaly to verify during the collapse: the momentum_phi/momentum_tor → name mapping is swapped between the two subtrees.

So the fix is not "pick one of long vs short" — it is a conditional canonical rule (locked in §Decisions): locus always; averaging prefix iff the FSA is not a no-op for that base.

A sibling instance: summary/composition/* yields 18 density-ratio names with inconsistent naming — ratio_of_<species>_density_to_electron_density vs <species>_fraction vs volume_averaged_<species>_..._to_..._electron_density. Same class (one quantity, many name shapes), lower priority.

§2 — Why it matters (motivation)

Multiple accepted names for one physical quantity is a first-order catalog defect: consumers cannot tell which name is authoritative, cross-references fork, and the same DD leaf appears under two or three ISN identifiers. It also breaks the ledger's implicit "one live name per quantity" expectation and inflates the catalog. The locus-stripped names are additionally wrong — a separatrix-averaged quantity that drops _at_plasma_boundary reads as a global quantity. This surfaced during sn-pedestal-source-recovery's blast-radius assessment (f-psr-002): the compose-grounding gap fixed there (WS-1) explains the locus-stripped variants, but the long/short multiplicity is a pre-existing naming-convention gap that the grounding fix does not resolve on its own.

The collisions are worse than the duplicates. A duplicate pair (long + short on the same leaf) merely forks references; a cross-subtree collision makes the name semantically ambiguous — toroidal_argon_velocity_at_plasma_boundary currently denotes both the local separatrix rotation and its flux-surface average, which are physically different numbers. Physics-relevant naming requires understanding, per base, whether the flux-surface average is a no-op; that classification must live in the vocabulary, not in per-name judgement calls, or the generation pipeline will keep re-minting the wrong forms (§4).

§3 — The rule + approach

The canonical form is a conditional rule (locked below), applied per base:

  • Locus always. Every name in the family carries _at_plasma_boundary. The ~6 locus-stripped names are wrong regardless of base class.
  • Prefix iff non-no-op. flux_surface_averaged_ appears iff the base is NOT constant on a flux surface. Which bases count as flux functions is the flux-function-base-list decision (open below); the definitional set is safety_factor, magnetic_shear, the flux labels (psi/rho_tor/rho_tor_norm position leaves), and pressure.

Execution, once the base list is locked and the §4 guards are in place:

  • Flux-function leaves — the local and averaged DD leaves attach to ONE shared name <q>_at_plasma_boundary (two sources → one StandardName, as poloidal_magnetic_flux_at_plasma_boundary already does). Supersede any flux_surface_averaged_-prefixed variant onto the shared name; attach detached sources deterministically (bind_recovery_sources-style targeted attach, never a --focus recompose).
  • Surface-varying leaves — the averaged leaf's canonical name keeps the prefix: flux_surface_averaged_<q>_at_plasma_boundary. Supersede the short duplicate on the averaged leaf onto it — UNLESS the short name is also produced by the local separatrix/* leaf (a collision, not a duplicate): then the short name stays as the local leaf's name and only the averaged source is detached/re-attached to the prefixed name. Check each short name's full source set before superseding; superseding a collided name would steal the local leaf's identity.
  • Add the missing locus — locus-stripped names rename to their canonical target via the rename cascade, or supersede onto an existing canonical sibling.
  • Verify the momentum swap — the momentum_phi/momentum_tor → name mapping is inverted between the two subtrees; resolve against DD semantics (the phi/tor pair is a DD v3/v4 alias set) while touching these leaves.
  • Generalise — the same conditional rule governs other flux-surface reduction contexts (summary/volume_average/* is a volume average — different operator, same no-op question for flux functions is NOT applicable there since a volume average of a flux function is not a no-op; scope it explicitly), pedestal fits, and maximum_over_flux_surface/minimum_over_flux_surface (max = min = value for a flux function — equally banned on flux-function bases).
  • Composition family — normalise the 18 summary/composition/* density-ratio names to one shape (lower priority; separate decision).

Mechanism: the deterministic rename/supersede cascade (graph_ops._execute_rename_cascade / sn run --rename OLD:NEW --include-accepted), NOT a --focus recompose — a recompose mints a fresh drafted variant and leaves the accepted duplicates, worsening the multiplicity (demonstrated in sn-pedestal-source-recovery). Verify 0 orphans / 0 desyncs and that every leaf resolves per the conditional rule.

§4 — Persistent fixes: vocab / grammar / prompt / grouper (land BEFORE the collapse)

Per-name renames alone do not survive regeneration — the pipeline would re-mint the wrong forms. The rule is encoded at four levels (work item f-bam-002, prerequisite for f-bam-001):

  • Vocab (ISN) — add a per-base flag constant_on_flux_surface: true to imas_standard_names/grammar/vocabularies/physical_bases.yml (schema comment + entries per the flux-function-base-list decision). This makes "FSA is a no-op for this base" machine-readable instead of a per-name judgement call.
  • Grammar (ISN) — a validation gate rejecting the flux-surface reduction operators (flux_surface_averaged, maximum_over_flux_surface, minimum_over_flux_surface) applied to a base flagged constant_on_flux_surface — the FSA-of-an-FSA class. Wire it where inherently_dimensional is already enforced (vocabulary validator / model audit), with tests. Names violating the gate fail composition, so the composer can never mint them again.
  • Prompts (imas-codex) — llm/prompts/sn/generate_name_dd.md's path-rule row for separatrix_average currently mandates the prefix unconditionally ("The name MUST carry the flux_surface_averaged transformation prefix … NEVER drop it") — replace with the conditional rule (prefix iff the base is not flagged constant_on_flux_surface; flux-function leaves share the local leaf's name). generate_name_system.md's averaging-operator bullet gains the no-op guard. review.md gets the matching review check.
  • Groupers / automatic parents (imas-codex) — reviewed 2026-07-10. standard_names/derivation.py::derive_edges already emits the right automatic parent: flux_surface_averaged_X_at_plasma_boundary → HAS_PARENT (operator_kind='unary_prefix') → X_at_plasma_boundary, and the parents.py admission gate (Clause A: structural specificity) correctly admits the locus-bearing parent. Gap: standard_names/harmonize.py::_FAMILY_OPERATOR_KINDS = (projection, qualifier, coordinate, locus) excludes unary_prefix, so the local/averaged pair never assembles into one docs family and their descriptions drift independently. Extend family eligibility to the averaging reduction operators (or record why not); standard_names/families.py (vector-component families for generation) is unaffected. A graph-side audit in standard_names/audits.py flags any live name violating the gate so regressions surface in sn run.

§ Decisions

What is the canonical form for a flux-surface-averaged quantity evaluated at the plasma boundary (separatrix)?

The locus and the averaging operator are orthogonal: _at_plasma_boundary pins WHICH flux surface (the boundary is one of a continuum, so the locus always carries weight), while flux_surface_averaged_ states a reduction that is only meaningful when the base varies over the surface. The DD encodes this with parallel separatrix/ (local) and separatrix_average/ subtrees, and the graph shows 15 live cross-subtree collisions from locus-only naming. For flux-function bases (safety factor — already a flux-surface quantity, so averaging it is an FSA of an FSA; magnetic shear; flux labels; pressure) FSA is a no-op and one shared name serves both leaves. Prefix iff non-no-op.

Which bases are flagged constant_on_flux_surface (FSA is a no-op → no averaging prefix, one shared name for the local and averaged leaves)?

Canonical shape for the summary/composition density-ratio names (ratio_of_X_density_to_electron_density vs X_fraction vs volume_averaged_…)?
