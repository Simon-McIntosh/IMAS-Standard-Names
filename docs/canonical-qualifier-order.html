§1 — The problem: qualifier order is not canonical

The ISN grammar specification (imas_standard_names/grammar/specification.yml, canonical_pattern) fixes the order of the named qualifier sub-kinds: aggregation → orbit → population → subject. It does not assign the open generic qualifier class (the 161 tokens in qualifiers.yml: major, external, absorbed, …) a position relative to the named sub-kinds or to each other. The parser preserves whatever order a name was written in (render_qualifiers emits qualifiers in parse order), so the catalogue has drifted into inconsistent ordering.

Measured across the live catalogue (1364 names): 1043 carry ≥1 qualifier, 477 carry ≥2. Under the natural rule "named sub-kinds in spec order, then generics", 104 names (7.6% of the catalogue) violate it — a generic qualifier sits before a named sub-kind. The headline evidence: subject → generic occurs in 79 names, but generic → subject occurs in 70 — the same two roles in the opposite order, so the ordering is genuinely not being maintained.

This surfaced in the catalog SPA's Grammar composer: when a user multi-selects qualifiers there is no canonical order to render them in, and seeded names can only be shown in their (possibly non-canonical) parsed order. The fix is upstream — a canonical order defined and enforced at generation time, with the catalogue migrated to it.

§2 — Define the canonical order

Extend the canonical pattern to give the generic qualifier class a defined position. The named sub-kinds keep their order; the open question (see Decisions) is where generics sit relative to the named sub-kinds and whether generics have an inter-order (e.g. by the 12 normalized categories from qualifier-vocabulary-categorization: transport → source → geometry → … ). The output of this section is a single, total order over every qualifier token, written into specification.yml.

§3 — Enforce at generate time

Make the renderer canonical: compose/render_qualifiers (imas_standard_names/grammar/render.py) must emit qualifiers in the locked order regardless of input order, and the validator must reject a name whose qualifiers are out of canonical order (so a non-canonical name can never be (re)introduced). Round-trip then means parse→canonical-compose, and any name that changes is a §5 migration target.

§4 — Template the order into the LLM name-generation prompts

The name-generation prompts (imas-codex sn generation; ISN get_grammar_context surfaces the canonical pattern + segment-order constraint to LLM consumers) must template the qualifier order explicitly so generating agents emit canonical names first time. Update _build_canonical_pattern / _build_segment_order_constraint to include the generic-qualifier ordering, and verify the codex generation prompt renders it.

§5 — Migrate the non-canonical catalogue names

Rename the ~104 violating names to canonical order. We are in the RC phase, so renames are acceptable. The migration runs through imas-codex (sn export/sn import — the catalogue YAML is generated, not hand-edited) with a deprecation/alias trail where consumers may reference old names. Verify the whole catalogue round-trips canonically afterward (build_site_dataset, full parse→canonical-compose).
