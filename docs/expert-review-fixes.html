§1 — Overview & method

A domain expert reviewed the generated Standard-Names (SN) catalog and left twelve comments spanning species spelling, charge-state aggregation, locus definitions, base-name precision, unit rendering, a suspected upstream unit bug, and a broken-equation report. This plan is the root-cause triage of that review: each comment was traced to its true origin in code, prompt, grammar/vocabulary, the generated catalog, or the upstream IMAS Data Dictionary (DD) — before any correction was proposed. No catalog names or code were changed while producing this plan; it is a verification-and-planning artifact.

The work lives across four surfaces, and every proposed fix is routed to the narrowest one that owns the defect:

  • ISN — imas-standard-names (this repo): the grammar engine and vocabulary (locus_registry.yml, physical_bases.yml, subjects.yml, qualifiers.yml, the parser, the unit format).
  • imas-codex: the sn generation pipeline — LLM compose/describe prompts (imas_codex/llm/prompts/sn/*.md), family harmonization, and the sn edit/sn run tooling.
  • ISNC — imas-standard-names-catalog: the generated names + descriptions, where sn edit and regeneration land.
  • DD — the upstream IMAS Data Dictionary: authoritative units and identifiers; defects here need an issue filed, not a local patch.

Guiding rule (from the request): a single-name blemish is fixed with sn edit; anything with a global root cause is fixed in the pipeline itself — code, schema, vocabulary, or prompt — so that regeneration keeps the fix. Ten of the twelve comments turned out to have a global root cause; only one is a per-name convention decision and one is already fixed.

Headline finding: the twelve surface comments collapse to six cross-cutting root causes. Fixing the six mechanisms — rather than editing individual names — corrects the reported symptoms and the much larger class of names that share each mechanism.

§ Decisions

C1 — Should compound species (deuterium-tritium, isotopes) use a hyphen, or keep the underscore?

Reassessed after review pushback. The parsing objection is RETRACTED: the pairs are enumerated vocab tokens (subjects.yml:65-67) and the parser splits only on '_' (parser.py:755), so a hyphenated 'deuterium-tritium' is a single atomic token — not a new delimiter, and it would even remove the token-internal-join vs grammar-segment longest-match ambiguity that '_' creates (the same class of ambiguity behind canonical-qualifier-order's shadowing followups). The real reasons to keep '_' are convention-level, not mechanical: (1) the published style guide (docs/guidelines.md:566, docs/development/style-guide.md:17) deliberately forbids hyphens ('flux-loop' rejected), CF-aligned; (2) '-' is already reserved for the machine/file-id namespace (docs/tokamak-parameters.md); (3) every other multi-word indivisible token (magnetic_axis, internal_transport_barrier, scrape_off_layer) uses '_', so hyphenating only species pairs is a new orthographic inconsistency; (4) consistency would cascade to isotopes already in vocab (helium_3, helium_4, methane_carbon_13 -> helium-3, carbon-13), touching accepted names; (5) CF/identifier portability. Net: keep underscore at MODERATE confidence. Flip only as a deliberate, DOCUMENTED convention change — hyphen = intra-token compound joiner applied consistently to compound species AND isotopes, guidelines/style-guide updated, machine-name namespace concern addressed — gated on a parse/compose/name-link/graph-key/SPA round-trip spike. Interacts with C12 (the D-T pair also becomes a reaction-channel qualifier there).

C5 — How to consolidate the pedestal locus?

Recommended: the first. By edge convention "the pedestal value" is the pedestal-top value (safe to rename ~value quantities), but maximum-gradient and flux-position quantities legitimately sit at the mid-pedestal (DD pedestal_fits/*/d_dpsi_norm_max_position , summary/local/pedestal/position/psi ) and must not be relabelled "top". A blanket rename would mislabel those.

C6 — Add a new base or rename the existing one?

Recommended: add a new base. The family is the only current consumer, so an in-place rename is technically safe, but adding a specific base keeps the generic coefficient slot free for future non-limiter coefficients and is the cleaner grammar.

C12 — How to make the fusion neutron-flux names precise?

Recommended: Option A. It mirrors the DD's neutron_fluxes/<pair> hierarchy 1:1, unifies with the existing neutron_flux_due_to_* names, and disambiguates the two D-D branches. Cost: parser positional disambiguation so the pair can be subject (density) or qualifier (neutron flux). Option B duplicates the neutron token across segments and produces awkward order.

C7 — How to make unit rendering consistent?

Recommended: ban-inline. The unit: field is already uniform and the prompt already forbids restating units in prose; the defect is purely legacy prose. A refine pass + a lint guard is simpler than a renderer and matches the existing rule. (A Unicode renderer could be added later purely for pill aesthetics, independent of this fix.)

C4·fsa — Scope of the separatrix pairing fix?

Recommended: velocity + density with the invariant. The gap is identical for both families and the reviewer's "for all quantities at the separatrix" is systematic; adding the invariant prevents the pipeline from shipping half a DD pair again.
