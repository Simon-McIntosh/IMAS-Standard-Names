§1 — Problem

The provenance-ledger rebuild drove orphans to zero, but 22 live names (0.8% of 2753) landed on source_type='manual' — the conservative "no deterministic anchor" fallback. That label is wrong: these names have a real DD origin, so each must carry a proper StandardNameSource(dd) — a first-class source node with FROM_DD_PATH to its IMASNode and PRODUCED_NAME to the name, exactly like every pipeline-generated name. They only became sourceless because of how they were created (§3), not because their DD leaf is missing (§2).

§2 — The DD leaves exist; the names must own a source

The 16 toroidal_<species>_velocity_at_pedestal_top names correspond to real DD leaves — the φ (toroidal) velocity of each species at the H-mode pedestal top. The recovery is to extract that leaf as a StandardNameSource(dd) and attach it, not to record a display-time "lookup":

φ = toroidal · summary/local/pedestal = pedestal top · one DD leaf per species (argon … tungsten, xenon). Extracting the leaf gives a real source node — the name owns its provenance, it is not "mapped" at read time.

The remaining ~6 need a per-name DD lookup: toroidal_plasma_momentum_at_pedestal_top, poloidal_magnetic_flux_at_pedestal_top, normalized_poloidal_magnetic_flux_at_pedestal_top, toroidal_diamagnetic_magnetic_flux_at_inside_flux_surface, flux_due_to_thermal_fusion, length_of_iron_core_segment, lower_energy. Each gets a real StandardNameSource from its DD leaf; any with genuinely no DD leaf is a naming question, not a provenance one, and is surfaced for review — never left silently sourceless.

§3 — How were these names created? (root cause)

Traced conclusively. The 22 carry origin='catalog_edit', catalog_commit_sha=a2f8831, model=None, imported in one batch (2026-07-04) into edge_plasma_physics.yml (+ a couple of other domains). They were hand-added to the catalog YAML (to satisfy a review request for per-species pedestal velocities) and then brought into the graph by the bulk import (run_import), which created the StandardName node but never created a StandardNameSource. That was the fault.

Confirmed this is the only path that could produce a sourceless name — and it is now closed:

  • sn edit cannot create names. apply_edit fetches the target and returns blocked if it does not already exist — it only edits existing names. Not the fault.
  • sn merge cannot create names. An unmatched catalog entry (no graph node by id) is reported as unmatched, never created.
  • The pipeline always sources. GENERATE_NAME composes a name from a StandardNameSource(dd) — a name cannot be born without one.
  • The bulk import — the one path that created sourceless nodes — was removed (provenance-ledger, commit 4b949931). Restore is now the diff-by-id reconciler (matches by id, no node creation); acceptance is sn merge.

So the fault is already closed: no remaining path can create a name without a source. The 22 are the residue left behind. The one systemic gap: the DD-extract evidently does not surface summary/local/pedestal/velocity_phi/<species>/value as extractable sources — else the pipeline would have composed and sourced these names instead of a human hand-adding them to the catalog. Closing that gap (WS-B) is how a future review request for such quantities is satisfied the right way.

  • WS-B · Close the extract gap.   • WS-C · Guardrail. The ledger invariant already reports orphans every sn run and gates release (Gate A). Confirm that if any name ever lands without a source again, it is surfaced for review rather than silently accepted.
  • Verify: the 22 carry real StandardNameSource(dd) nodes; manual count drops to only names with genuinely no DD leaf (surfaced as a naming question); 0 orphans / 0 desyncs preserved; name/description/docs/stage untouched.
