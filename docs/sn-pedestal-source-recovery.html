§1 — Problem (and what it is NOT)

After the provenance-ledger rebuild, 22 live names carry source_type='manual'. Investigation shows this is not a lost-provenance problem and not a fabrication problem — it is a timing / premature-binding problem. Their real dd sources exist and are simply not composed yet: the SN pipeline is only half-run, and those sources sit in the GENERATE_NAME backlog. The rebuild, run on that half-composed graph, mistook "not-composed-yet" for "no anchor" and applied the conservative manual fallback.

§2 — Evidence: the sources are extracted, pending composition

The graph holds 996 extracted, unlinked dd StandardNameSource nodes (1013 in extracted status overall) — the live GENERATE_NAME queue. A source in this state has been extracted from a DD leaf but not yet composed into a name; it is claimable by the next sn run. All 16 summary/local/pedestal/velocity_phi/<species>/value sources are in this queue (status='extracted', no PRODUCED_NAME, produced_sn_id=null), and each is node_category=quantity with docs — they pass the extract gate cleanly. So the DD-extract is working; the pipeline just hasn't reached these yet.

The names already exist (imported from the catalog); once GENERATE_NAME composes the pending sources, consolidation attaches them to those names — giving each its real dd source with no manual step.

§3 — How the names were created, and why 'manual' was premature

The 22 are origin='catalog_edit' (21) + one origin='pipeline', imported 2026-07-04. They entered the graph as catalog entries; their DD sources were extracted separately and are still in the GENERATE_NAME queue. No creation path leaves a name permanently sourceless: sn edit only edits existing names (apply_edit blocks a missing target), sn merge reports unmatched entries rather than creating them, and the pipeline always sources. The one historical path that created sourceless nodes — the bulk import — is removed.

The rebuild's fault was ordering: it ran while ~1000 dd sources were still uncomposed, and its orphan definition ("live name with no PRODUCED_NAME") counted names whose source is merely pending. For those, the conservative tree found no recovery map entry and no HAS_PARENT, so it minted a manual source — pre-empting the pipeline.

§ Decisions

Recover by draining the pipeline (compose the pending sources), or by a targeted deterministic attach of the known source→name pairs?

LOCKED (Simon McIntosh, 2026-07-09): drain via a SCOPED `sn run --focus` over the confirmed DD leaves (15 summary/local/pedestal/velocity_phi/<species>/value + summary/local/pedestal/momentum_phi/value) — the canonical compose/consolidation path attaches the real dd StandardNameSource to each existing accepted name. Explicitly scoped with --focus, NOT a full ~1000-source backlog drain (out of scope for now, per lead), and NOT the deterministic targeted-attach.

How to stop the rebuild manual-binding pending-source names (WS-B)?

LOCKED (Simon McIntosh, 2026-07-09): exclude-pending. The rebuild cascade gains a guard before the manual fallback: an orphan with a claimable pending/extractable DD source is NOT manual-bound (left for the pipeline to source). Correct regardless of pipeline state; regression-tested. Preferred over the operational gate-on-drain ordering rule which leaves no code guard against a future half-run.
