§2 — DECISION: cylindrical axis-token naming (radial/toroidal/vertical vs r/phi/z)

Lead question: given the Cartesian family is x, y, z, should the cylindrical machine axes be r, phi, z for symmetry, rather than the current words radial, toroidal, vertical?

Current scheme (after rc60)

FrameHorizontal axesThird (machine-vertical) axisWhat the tokens are
Device-local Cartesianx, yz Arbitrary mounting axes — you cannot say which physical direction x points without the device's survey
Machine cylindrical (R, φ, Z)radial, toroidalvertical Physically named directions — major-radius outward, around the torus, machine up
Plasma / field directionspoloidal, parallel, perpendicular, normal, tangential, binormal (+ normalized_*)— Physically named directions — no natural single-letter form exists

The organizing principle already in place is: letters name arbitrary local axes; words name physically-meaningful directions. This is the CF-standard-names convention (eastward, upward) and it is why z (Cartesian, arbitrary) and vertical (machine, physical) are distinct tokens for the same physical axis — the token also encodes which frame the name lives in, which is exactly the anti-collapse property the rc60 work was built around (no token appears in two frames). Catalog usage today: 49 vertical_*, 21 radial_*, 11 toroidal_*, 5 poloidal_*, and 5+5+1 Cartesian x/y/z_*.

Options

Option A — keep words for physical directions (RECOMMENDED). No change: x, y, z for device-local Cartesian, radial, toroidal, vertical (and poloidal/parallel/…) for physical directions. Pros: coherent organizing principle; frame-marked (preserves the collapse-guard); self-documenting; honors the D.3 senior review (2026-04) that ruled toroidal_component canonical and declared velocity_phi forbidden, velocity_tor deprecated; zero churn (just shipped). Con: the shared vertical axis carries two tokens (z / vertical), and the letter-vs-word split between frames looks asymmetric on the surface.

Option B — rename cylindrical to r/phi/z for symmetry (NOT recommended). Pros: superficial 3-token parallelism with x, y, z. Cons, decisive: (1) reintroduces z into the machine frame, so z lives in both frames again — weakening the frame-marking/collapse-guard the whole rc60 effort established; (2) only 3 of the ~9 physical directions have letters — poloidal, parallel, perpendicular, normal, tangential, binormal do not — so you get r, phi, z beside poloidal, parallel, …, a worse mix than today; (3) reverses the D.3 senior ruling; (4) phi as a direction token collides conceptually with the toroidal angle φ we just standardized in descriptions; (5) ~81 catalog names churn plus prompt/grammar/doc changes.

Option C — keep words canonical, but honor the input aliases (RECOMMENDED companion to A). coordinate_axes.yml already declares aliases (radial:[r], toroidal:[phi, tor], poloidal:[pol]), but parser.py loads only the vocab keys (axes=frozenset(axes_reg.axes)) and never consumes the alias lists — verified — so the aliases are dead metadata. Fixing the parser to honor them lets DD paths and source authors write r/ phi on input (the ergonomic symmetry itch) while the stored canonical name stays word-based. No catalog churn, honors D.3, keeps frame-marking; the only work is a genuine parser bug fix (the same class of latent bug found for the seven dead semantic checks).

Recommendation: A + C. Keep the letters-for-local / words-for-physical principle, and make the declared input aliases actually work. B trades a coherent, collapse-safe, precedent-backed scheme for surface symmetry that it cannot even deliver (the flux directions have no letters). This is the lead's call — recorded as the open decision below.

§3 — Tooling: sn run --reset-to ignores --focus (confirmed unfixed)

Confirmed 2026-07-05, still live. reset_standard_names / clear_standard_names (graph_ops.py) accept only source_filter (dd/signals) and ids_filter (IDS-name prefix) — no dd-path list — and the CLI's --reset-to branch (cli/sn.py ~L1471) never threads the --focus path list into the reset call. So sn run --reset-to extracted --focus <paths> resets far more than the focused paths, and with --include-accepted it cleared 1863 accepted names graph-wide during the rc60 session (recovered via sn import). This is a live footgun.

Options. (a) Thread a dd-path allowlist into reset_standard_names/ clear_standard_names and wire --focus so --reset-to … --focus scopes to exactly those paths. (b) Make --reset-to combined with --focus or --include-accepted a hard error (fail fast on the dangerous combo). (c) Leave the code, keep the documented surgical-DETACH DELETE-after-dry-run workaround. Recommendation: (a) + (b) — give reset real path scoping AND guard the specific combo that caused the incident. Recorded as the open decision below.

§ Decisions

Cylindrical/physical axis tokens: keep words, switch to r/phi/z, or keep words + honor input aliases? (§2)

Recommendation A+C: letters name arbitrary local axes, words name physical directions (CF convention, D.3 senior ruling, frame-marking / collapse-guard). B cannot deliver symmetry (poloidal/parallel/… have no letters), re-collides z across frames, reverses D.3, and conflates the phi direction with the φ angle. C fixes the dead alias metadata so r/phi are accepted on input without catalog churn.

How to fix sn run --reset-to ignoring --focus? (§3)

Recommendation scope-and-guard: path scoping makes focused resets actually work; the guard prevents the accepted-name-wipe footgun that cost a full recovery cycle.
