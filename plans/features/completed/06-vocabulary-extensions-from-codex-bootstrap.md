# Feature Plan: Vocabulary Extensions from Codex Bootstrap

## Problem

The imas-codex standard names bootstrap pipeline generated 342 StandardName entries across 8 IDS
(equilibrium, core_profiles, edge_profiles, summary, mhd, transport_solver_numerics, pf_active,
wall). Automated grammar analysis reveals **38 names** that could be better-structured if ISN
vocabulary were extended, plus **9 names** that fail ISN parse entirely.

The grammar is already more capable than the compose pipeline exploits â€” transformations, binary
operators, and processes all work. The gaps are in **missing tokens within existing segments**,
not in grammar architecture.

## Proposed Vocabulary Extensions

### Priority 1: Transformation Tokens (20 names affected)

**Current transformations (4):** `square_of`, `change_over_time_in`, `logarithm_of`, `inverse_of`

**Proposed additions:**

| Token | Evidence | Names Affected | Rationale |
|-------|----------|---------------|-----------|
| `time_derivative_of` | 11 names use this pattern | 11 | More intuitive synonym for `change_over_time_in`. Physicist-facing term. Consider as alias or replacement. |
| `maximum_of` | `maximum_charge_state`, `maximum_ion_charge_state` | 3 | Extremum over a population or profile |
| `minimum_of` | `minimum_ion_charge_state` | 1 | Extremum complement |
| `volume_averaged` | `volume_averaged_ion_temperature` | 1 | Spatial averaging transformation |
| `flux_surface_averaged` | Common in transport quantities | 1 | Standard tokamak operation |
| `normalized` | `normalized_toroidal_beta` | 1 | Dimensionless normalization |
| `second_derivative_of` | `second_radial_derivative_of_*` (3 names) | 4 | Higher-order derivative. Note: may need compound with spatial direction. |

**Design question:** Should `time_derivative_of` replace or alias `change_over_time_in`?
`change_over_time_in` follows CF-convention style but `time_derivative_of` is mathematically
standard. All 11 names generated by the compose LLM used `time_derivative_of`, not
`change_over_time_in`.

**Design question:** Spatial derivatives (`radial_derivative_of`, `second_radial_derivative_of`)
need a mechanism to specify the differentiation variable. Current grammar doesn't support
parameterized transformations. Options:
1. Full tokens: `radial_derivative_of`, `second_radial_derivative_of` (simple but grows the vocab)
2. Compound pattern: `derivative_of_X_with_respect_to_Y` (uses binary operator architecture)
3. Defer: Keep as compound physical_base for now

### Priority 2: Process Tokens (9 names affected)

**Current processes (26):** bootstrap, ohmic, radiation, etc.

**Proposed additions:**

| Token | Evidence | Names Affected | Rationale |
|-------|----------|---------------|-----------|
| `non_inductive` | `non_inductive_current` | 1 | Standard plasma physics concept: current not driven by induction |
| `gas_injection` | 7 gas injection names | 7 | Material injection process; needed for fueling quantities |
| `pellet_injection` | Not yet in bootstrap data but expected | 0 | Natural companion to gas_injection |

**Design note:** `bootstrap_current` currently parses as base-only because the LLM wrote it as
`bootstrap_current` not `plasma_current_due_to_bootstrap`. This is a **compose prompt issue**, not
an ISN gap â€” the grammar already handles `bootstrap` as a process. Solution is prompt improvement
in codex, not ISN change.

### Priority 3: Subject Tokens (2 names affected)

**Current subjects (9):** electron, ion, deuterium, tritium, helium, impurity_species, fast_ion,
neutral, runaway_electron

**Proposed additions:**

| Token | Evidence | Names Affected | Rationale |
|-------|----------|---------------|-----------|
| `alpha_particle` | Needed for burning plasma / summary IDS | 0 (needed) | ITER burning plasma regime requires alpha particle tracking |
| `thermal` | Qualifier for thermal vs fast populations | 1 | `thermal_deuterium_neutron_rate` â€” thermal subpopulation |

**Design note:** `thermal` may be better as a new segment type (qualifier) rather than a subject,
since it modifies another subject: `thermal_electron`, `thermal_ion`. This needs grammar design
discussion.

### Priority 4: Parse Failure Fixes (9 names)

| Name Pattern | Error | Root Cause | Fix |
|-------------|-------|------------|-----|
| `*_component_of_diamagnetic_velocity` (6) | component/coordinate conflict | `diamagnetic` parsed as both component and coordinate | ISN: clarify `diamagnetic` disambiguation; Codex: rename to avoid conflict |
| `electron_toroidal_component_of_velocity_*` (2) | Component must precede subject | Grammar ordering violation | Codex: compose prompt fix (reorder segments) |
| `toroidal_component_of_velocity_due_to_charge_exchange` (1) | Generic physical_base `velocity` | Needs qualification | Codex: use `plasma_velocity` instead |

The diamagnetic issue (6 names) is fundamental â€” `diamagnetic` appears in both the component
and coordinate vocabularies. When used as `radial_component_of_diamagnetic_velocity`, the parser
sees `diamagnetic` as coordinate, conflicts with `radial` as component. **This requires ISN
grammar disambiguation, not just vocab extension.**

## Surfacing Mechanism

### Recommended: CLI Extension + MCP Tool

1. **`sn gaps` CLI command** â€” query graph for grammar issues, group by gap type
2. **`list_standard_names(review_status='vocab_gap')`** â€” MCP tool already supports this filter
3. **`sn gaps --export yaml`** â€” export in format suitable for ISN GitHub issues

**Not recommended:** Agent-only interaction (gaps need to be persistent and reviewable by humans).

### Implementation in Codex

The `vocab_gap_detail` field exists on StandardName but is never populated. Phase 1A of Plan 26
(SN Pipeline Quality Iteration) addresses this wiring. Once wired:

1. Compose worker detects when it uses base-only as a fallback
2. Review worker assesses grammar richness and flags base-only stuffing
3. Both write `vocab_gap_detail` JSON to the StandardName node
4. `sn gaps` CLI aggregates and formats for ISN issue filing

## Implementation Order

### Phase 1: Codex-Side (no ISN changes needed)

1. Wire vocab gap persistence (Plan 26, Phase 1A)
2. Improve compose prompts to use existing grammar (e.g., `due_to_bootstrap` not `bootstrap_current`)
3. Add `sn gaps` CLI command
4. Regenerate 9 parse-failure names with corrected prompts

### Phase 2: ISN Vocabulary Extension (requires ISN release)

1. Add transformation tokens: `maximum_of`, `minimum_of`, `volume_averaged`,
   `flux_surface_averaged`, `normalized`
2. Add process tokens: `non_inductive`, `gas_injection`, `pellet_injection`
3. Discuss `time_derivative_of` vs `change_over_time_in` aliasing
4. Discuss `diamagnetic` component/coordinate disambiguation
5. Discuss `thermal` qualifier segment

### Phase 3: Codex Regeneration (after ISN release)

1. `pip install --upgrade imas-standard-names`
2. `sn reset --status vocab_gap --to drafted`
3. `sn generate --source dd --force` on affected IDS
4. `sn review --re-review`
