## Phase 1: Grammar Ambiguity Resolution (FIRST — architectural risk)

The codex VocabGap graph records 9 `grammar_ambiguity` entries where the parser cannot
unambiguously assign tokens to segments. These must be resolved before adding new vocabulary,
because new tokens could introduce the same class of bug.

### Known Ambiguous Tokens

| Token | Conflict | Impact |
|-------|----------|--------|
| `diamagnetic` | Component ∩ physical_base prefix | 6 names fail parse (`radial_component_of_diamagnetic_velocity`) |
| `ion` | Subject ∩ physical_base prefix | `ion_temperature` parses correctly, but `ion_cyclotron_*` triggers segment confusion |
| `neutral` | Subject ∩ physical_base prefix | `neutral_density` vs `neutral_beam_*` ambiguity |
| `vertical` | Component ∩ position qualifier | `vertical_position` vs `vertical_component_of_*` |
| `ion_state` | Subject compound vs physical_base | Charge state quantities |
| `r` | Major radius vs cylindrical R | Geometric coordinate ambiguity |
| `strike_point` | Position vs geometry target | Divertor topology |
| `outline_geometry_object_distinction` | Geometry vs object | Boundary representations |
| `measured_vs_reconstructed_distinction` | No grammar segment for provenance | Design gap |

### Root Cause Analysis

The parser's greedy prefix matching consumes tokens from earlier segments (component, coordinate,
subject) before checking whether a longer compound physical_base match exists. The same pattern
causes ambiguity for `diamagnetic`, `ion`, `neutral`, and `vertical`.

### Fix: Longest-Match Physical Base Priority

**Files:** Parser module (investigate exact location at implementation time)
**Tests:** `tests/test_grammar_ambiguity.py` (new file)

**Approach:**
1. When a token matches both a closed-vocabulary segment AND is a prefix of a known or open
   physical_base, prefer the physical_base interpretation
2. The component/coordinate/subject segments are closed vocabulary — all tokens are known
3. Physical_base is open vocabulary — but compound bases like `diamagnetic_velocity`,
   `neutral_beam_power`, `ion_cyclotron_frequency` are recognizable patterns

**Design decision needed:** Should disambiguation be:
- **(A) Parser heuristic** — longest token match wins (simple, may have edge cases)
- **(B) Explicit exception list** — tokens with known dual roles get special handling
- **(C) Compose-time canonicalization** — the compose LLM avoids ambiguous forms

**Recommended:** (A) with (B) as fallback. Longest-match is the standard approach in grammar
parsers. Maintain an explicit test for each known ambiguous token to prevent regressions.

**Tests:**

```python
# tests/test_grammar_ambiguity.py
def test_diamagnetic_velocity_not_split():
    """diamagnetic_velocity is compound physical_base, not component + base."""
    name = "radial_component_of_diamagnetic_velocity"
    parsed = parse_standard_name(name)
    assert parsed.component == "radial"
    assert "diamagnetic_velocity" in parsed.physical_base

def test_ion_temperature_subject_not_base():
    """ion is subject in ion_temperature, not start of physical_base."""
    name = "ion_temperature"
    parsed = parse_standard_name(name)
    assert parsed.subject == "ion"
    assert parsed.physical_base == "temperature"

def test_neutral_density():
    """neutral is subject in neutral_density."""
    name = "neutral_density"
    parsed = parse_standard_name(name)
    assert parsed.subject == "neutral"
    assert parsed.physical_base == "density"

def test_vertical_component():
    """vertical is component when followed by component_of."""
    name = "vertical_component_of_magnetic_field"
    parsed = parse_standard_name(name)
    assert parsed.component == "vertical"

def test_vertical_position():
    """vertical_position is a compound physical_base."""
    name = "vertical_position_of_magnetic_axis"
    parsed = parse_standard_name(name)
    assert "vertical_position" in parsed.physical_base or parsed.geometric_base is not None
```

**Canonicalization decision for `time_derivative_of` vs `change_over_time_in`:**

Both are valid. Adding both tokens (no alias mechanism needed). Document as synonyms — the
compose LLM in codex will converge on one form. The parser accepts either.
`change_over_time_in` is CF-convention style; `time_derivative_of` is mathematician convention.
Both describe the same transformation.

**Deferred (justified):**

| Token | Issue |
|-------|-------|
| `second_derivative_of` | Higher-order derivatives need design pattern for chaining |
| `derivative_with_respect_to_rho_tor` | Too coordinate-specific — compose as compound |
| `finite_element_coefficients` | Not a transformation — numerical method artifact. Codex classifier should skip these |
| `ratio_to_fuel` | Not a transformation — ratio to a reference value. Codex should use binary operator |

    parsed = parse_standard_name(name)

## Phase 3: Validation Enhancement

### None vs Dimensionless Unit Semantics

**Current state:** `_check_dimensionless_physical_quantity()` in `validation/semantic.py` warns
when physical quantities have unit `'1'` (dimensionless). But the None/dimensionless distinction
is not enforced.

**Rule:**
- `unit: None` → valid only for `kind: metadata` (identifiers, labels, enumerations)
- `unit: '1'` (dimensionless) → valid for `kind: scalar` or `kind: vector` (safety factor, beta)
- `unit: None` + `kind: scalar` → WARNING (likely should be `'1'` or a specific unit)

**Files:** `imas_standard_names/validation/semantic.py`
**Tests:** Extend `tests/test_unit_validation.py`

```python
def test_none_unit_with_metadata_is_valid():
    entry = create_standard_name_entry(name="...", kind="metadata", unit=None, ...)
    issues = run_semantic_checks("test_name", entry)
    assert not any("unit" in i.lower() for i in issues)

def test_none_unit_with_scalar_warns():
    entry = create_standard_name_entry(name="...", kind="scalar", unit=None, ...)
    issues = run_semantic_checks("test_name", entry)
    assert any("WARNING" in i and "unit" in i.lower() for i in issues)

def test_dimensionless_with_scalar_is_valid():
    entry = create_standard_name_entry(name="...", kind="scalar", unit="1", ...)
    issues = run_semantic_checks("test_name", entry)
    assert not any("dimensionless" in i.lower() and "ERROR" in i for i in issues)
```
